# 持续认证服务项目功能文档与流程分析

> 文档说明：本文件基于当前仓库代码实现进行分析，重点围绕“数据接收 → 数据存储/预处理 → 模型训练 → 在线推理 → 向 App 交互传输”这条主链路说明项目实际具备哪些功能、每个功能做了什么、逻辑是否合理，以及当前存在的主要问题。

## 1. 结论先行

从代码实现看，这个项目已经不是一个单纯的“接收手机传感器数据并落盘”的服务，而是一个**面向移动端连续认证（Continuous Authentication）的服务端原型系统**，其主链路是：

1. App 通过 gRPC 获取上传策略；
2. App 发起认证会话；
3. App 持续上传加密压缩后的传感器批次；
4. 服务端解密、解压、解析并保存原始数据；
5. 当原始数据达到阈值后，自动触发预处理与模型训练；
6. 当认证会话已建立且模型可用时，服务端对新批次做实时推理；
7. 服务端通过同一条 gRPC 流向 App 回传 `Ack` 和 `AuthResult`；
8. App 根据 `AuthResult` 决定是否提示风险、打断或锁屏。

项目的**总体架构是清晰的**，模块划分也比较明确：

- Web/HTTP 层：`src/main.py`、`src/api/*`
- gRPC 主交互层：`src/grpc_server.py`
- 加解密与压缩：`src/crypto/*`
- 原始数据存储：`src/storage/file_storage.py`
- 在线推理结果存储：`src/storage/inference_storage.py`
- 数据预处理：`src/processing/pipeline.py`
- 训练编排：`src/training/manager.py`、`src/training/runner.py`
- 在线认证：`src/authentication/manager.py`、`src/authentication/vqgan_inference.py`
- 离线策略搜索：`src/policy_search/*`
- 实际模型训练代码：`ca_train/*`

但是，代码深入分析后可以看到：

- **主流程能跑通，但仍然是“研究原型/工程验证版”，并非严格一致的生产级实现**；
- **协议定义、训练流程、在线推理、策略搜索之间存在若干严重不一致**；
- **HTTP 接口与 gRPC 接口已明显分叉，HTTP 更像遗留兼容接口**；
- **加密协议、策略文件、在线决策规则、离线最优策略之间存在逻辑错位**。

如果只从“原型功能是否完整”来看，这个项目已经具备较完整的端到端闭环；
如果从“协议严谨性、在线/离线一致性、生产可用性”来看，当前还有若干关键问题需要优先修正。

## 2. 项目定位与整体架构

### 2.1 项目目标

本项目的目标不是做传统用户名/密码认证，而是做**基于手机多传感器行为数据的持续身份认证**。它尝试利用加速度计、陀螺仪、磁力计等时序数据，训练出“合法用户”的行为模型，并在用户使用设备过程中持续判断当前操作者是否仍是本人。

### 2.2 实际上的身份粒度

虽然协议里同时存在 `device_id_hash` 与 `user_id_hash`，但从当前实现看，系统的主索引基本是 **`device_id_hash`**：

- 原始数据按 `device_id_hash` 建目录；
- 模型按 `device_id_hash` 建目录；
- 认证会话也按 `device_id_hash` 找模型；
- `SerializedSensorBatch.user_id_hash` 在在线主流程中几乎没有被真正使用。

这意味着：

- **当前系统本质上更像“按设备训练、按设备认证”**；
- 如果未来希望支持“一个用户多个设备”或“同一设备多用户”，当前设计还不够一致。

### 2.3 总体时序

下面是当前代码对应的真实主链路：

```text
App
 ├─(1) GetInitialPolicy ----------------------------> Server
 │                                                    └─返回上传/压缩/采样配置
 ├─(2) StartAuthentication -------------------------> Server
 │                                                    ├─若模型已就绪：建立认证会话
 │                                                    └─若模型未就绪：拒绝并提示数据不足或训练中
 ├─(3) StreamSensorData(DataPacket...) -------------> Server
 │                                                    ├─解密
 │                                                    ├─解压
 │                                                    ├─解析 protobuf 传感器批次
 │                                                    ├─保存原始包到 raw_data
 │                                                    ├─检查是否需要触发训练
 │                                                    ├─若认证会话存在：执行在线推理
 │                                                    ├─先返回 Ack
 │                                                    └─随后可能返回 AuthResult
 ├─(4) SendHeartbeat -------------------------------> Server
 │                                                    └─返回时间回显
 └─(5) ReportMetrics -------------------------------> Server
                                                      └─服务端仅接收并记录
```

## 3. 代码结构与职责分层

### 3.1 服务入口层

- `src/main.py`
  - 初始化日志；
  - 创建 FastAPI 应用；
  - 注册 HTTP 路由；
  - 启动 gRPC 服务；
  - 当 HTTP 和 gRPC 配置同端口时，自动关闭 HTTP，只保留 gRPC 单端口模式；
  - 探测 TLS 证书，支持 TLS 与 h2c 两种模式。

### 3.2 网络接口层

- `src/grpc_server.py`
  - 是当前 App 主交互入口；
  - 实现了 `GetInitialPolicy`、`StartAuthentication`、`StreamSensorData`、`SendHeartbeat`、`ReportMetrics`；
  - 管理数据接收、训练触发、在线认证与响应回传。

- `src/api/sensor_data.py`
  - 是 HTTP/JSON 风格接口；
  - 仍可接收加密压缩负载并写入 JSONL；
  - 但与 gRPC 主链路的协议格式、校验逻辑和下游流程已经明显分叉。

### 3.3 算法与数据层

- `src/processing/pipeline.py`
  - 负责原始数据转结构化训练数据；
  - 包括选 session、抽取传感器、重采样、划分 train/val/test、拼接 HMOG 攻击者样本、归一化、滑窗。

- `src/training/manager.py`
  - 负责判断某用户数据是否达到训练条件；
  - 并异步启动预处理与训练。

- `src/training/runner.py`
  - 负责调用 `ca_train/hmog_vqgan_experiment.py` 实际训练模型；
  - 训练结束后生成 `best_lock_policy.json`。

- `src/authentication/manager.py`
  - 负责建立在线认证会话；
  - 维护推理窗口状态、缓存模型、执行投票/K 连拒规则。

- `src/policy_search/runner.py`
  - 负责离线策略网格搜索；
  - 搜索投票参数 `(N, M)` 与阈值，输出 Pareto 前沿和最优策略。

- `ca_train/*`
  - 是真正的模型训练与离线推理实现；
  - 包含 VQGAN 模型、数据加载、窗口转张量、阈值计算、会话打断统计等研究代码。

## 4. 数据接收流程分析

本节分成两个入口：**gRPC 主流程** 与 **HTTP 遗留流程**。

### 4.1 gRPC 主流程

核心代码：`src/grpc_server.py`、`protos/sensor_data.proto`

#### 4.1.1 提供了哪些功能

1. `GetInitialPolicy`
   - 向 App 下发初始采集/上传策略；
   - 包括采样率、压缩算法、批大小阈值、上传速率等。

2. `StartAuthentication`
   - App 请求开始一次认证会话；
   - 如果模型已就绪，则接受；否则返回“数据不足”或“训练中”。

3. `StreamSensorData`
   - 双向流式收发；
   - App 连续上传 `DataPacket`；
   - 服务端按包返回 `Ack`，并在需要时返回 `AuthResult`。

4. `SendHeartbeat`
   - 心跳保活与时间回显。

5. `ReportMetrics`
   - 接收客户端聚合运行指标。

#### 4.1.2 每个包服务端实际做了什么

对每个 `DataPacket`，服务端执行顺序如下：

1. 读取 `device_id_hash`、`packet_seq_no`、`metadata` 等头信息；
2. 对 `encrypted_sensor_payload` 执行 **先解密、后解压**；
3. 将解压结果按 `SerializedSensorBatch` protobuf 解析；
4. 将原始包（以及解析后的 `sensor_batch`）写入 `data_storage/raw_data/<device>/session_<session>.jsonl`；
5. 若存储成功，异步检查该设备数据量是否达到训练阈值；
6. 若当前设备已经建立认证会话，异步启动在线推理；
7. 立即返回 `Ack`；
8. 推理完成后，再通过流返回 `AuthResult`。

#### 4.1.3 gRPC 接收流程的合理性

**合理的地方：**

- 使用 gRPC 双向流很适合高频小包持续上传场景；
- `Ack` 与 `AuthResult` 分离，使“接收成功”和“认证结论”可以异步返回；
- “先落原始包，再触发训练/推理”有利于保留审计材料；
- `StartAuthentication` 把“是否允许进入在线认证”前置，逻辑清晰；
- `SendHeartbeat` 与 `ReportMetrics` 为移动端运行态留出了观测接口。

#### 4.1.4 gRPC 接收流程存在的问题

**问题 1：协议定义与实际加密实现严重不一致。**

协议声明：

- `encrypted_sensor_payload` 是 Envelope/StreamingAEAD 密文；
- `encrypted_dek` 应该由服务器公钥解密；
- `dek_key_id` 用于选择密钥版本；
- `sha256` 用于校验。

但实际实现里：

- 服务器直接用 `settings.encryption_key` 推导出一个固定 AES-256-GCM 密钥；
- `encrypted_dek` 没有被使用；
- `dek_key_id` 没有被使用；
- `sha256` 没有被校验；
- 也没有真正的密钥轮换机制。

这说明：

- **当前实现是“固定对称密钥解密”**，不是 proto 注释所描述的“信封加密”；
- 协议层看起来设计成了生产方案，代码层却还是原型方案；
- 安全设计与实现脱节，是本项目最大的结构性问题之一。

**问题 2：`Ack.success` 只表示“是否存储成功”，不表示“是否成功解析/可用于训练推理”。**

当前代码中，即使：

- 解密失败；
- 解压失败；
- protobuf 解析失败；

只要原始包记录成功写入磁盘，`Ack.success` 仍可能为 `true`。

这会导致：

- App 误以为“服务端已成功接收并处理数据”；
- 但实际上该包可能根本没有进入训练/推理有效链路；
- 客户端与服务端对“成功”的语义不一致。

**问题 3：抗重放字段存在，但服务端没有真正使用。**

协议里有：

- `packet_seq_no`
- `sha256`
- `seq_no`

但在线 gRPC 处理里：

- 没有检查包序号是否递增；
- 没有去重；
- 没有校验哈希；
- 没有会话级防重放缓存。

这意味着：

- 当前“抗重放”更多停留在字段设计层面；
- 实际上仍可能接收重复包、乱序包、重放包。

**问题 4：`user_id_hash` 字段存在，但在线流程实际上按 `device_id_hash` 运行。**

这会导致“用户身份”和“设备身份”概念混合。当前逻辑没有真正消费 `SerializedSensorBatch.user_id_hash`。

**问题 5：支持的服务端指令类型比实际实现多。**

`ServerDirective` 定义了：

- `Ack`
- `PolicyUpdate`
- `KeyRotationNotice`
- `EmergencyStop`
- `AuthResult`

但当前流式实际会下发的，主要只有：

- `Ack`
- `AuthResult`

`KeyRotationNotice` 和 `EmergencyStop` 仅停留在 proto 设计上，尚未在服务逻辑中真正使用。

### 4.2 HTTP 遗留流程

核心代码：`src/api/sensor_data.py`、`src/validators/packet_validator.py`

#### 4.2.1 提供了哪些功能

- `POST /api/v1/sensor-data`
- Header 中携带：`X-Device-ID-Hash`、`X-Session-ID`、`X-Packet-Sequence`
- Body 是加密压缩后的 JSON 数据
- 服务端解密、解压、JSON 解析、Pydantic 校验、写 JSONL

#### 4.2.2 实际做了什么

HTTP 入口做的是：

1. 限制请求大小；
2. 读取请求体；
3. 解密、解压；
4. 将解压结果按 JSON 解析；
5. 用 `PacketValidator` 校验 Header 与包体是否一致；
6. 将 JSON 直接写入 raw_data；
7. 返回 `{status: ok}`。

#### 4.2.3 这条流程是否合理

作为“早期调试入口”或“兼容旧客户端入口”，这是合理的。

但是从整个项目当前状态看，HTTP 路径**并不是主链路**，因为：

- 它的包格式是 JSON；
- gRPC 主流程处理的是 protobuf `SerializedSensorBatch`；
- HTTP 路径没有接入认证会话管理；
- HTTP 路径没有触发在线推理；
- HTTP 路径没有显式接入训练管理器。

#### 4.2.4 存在的问题

**问题 1：HTTP 与 gRPC 两条链路的数据协议已经分叉。**

- HTTP：解压后是 JSON；
- gRPC：解压后是 protobuf binary；
- HTTP 校验器要求 `device_id_hash`、`session_id` 为整数；
- gRPC 主流程使用的是字符串型 hash / UUID session。

这说明 HTTP 更像历史遗留接口，而不是当前 App 真实使用的主入口。

**问题 2：HTTP 路径缺少与训练/推理闭环的完整联动。**

它能收数据，但没有和 gRPC 主流程那样自然地进入“会话认证 + 结果回传”的闭环。

**问题 3：时间戳校验硬编码到 2020~2030。**

`PacketValidator` 将时间戳限制在 2020 到 2030 之间，这对于长生命周期系统是不合理的：

- 到了 2030 年后，合法数据会被整体拒绝；
- 即使 2030 之前，这个规则也缺少业务依据与配置化能力。

## 5. 数据存储与目录组织分析

### 5.1 原始数据存储

核心代码：`src/storage/file_storage.py`

当前原始数据以 JSONL 方式落盘：

```text
data_storage/raw_data/<device_id_hash>/session_<session_id>.jsonl
```

每来一个包，追加一行 JSON。

存储内容包括：

- 包头字段；
- metadata；
- 解密/解压状态；
- 若解析成功则保存 `sensor_batch`；
- 若解析失败则保存原始加密载荷的 base64。

### 5.2 在线推理存储

核心代码：`src/storage/inference_storage.py`

在线认证相关数据单独落盘：

```text
data_storage/inference/<device_id_hash>/<session_id>/raw.jsonl
data_storage/inference/<device_id_hash>/<session_id>/results.jsonl
```

其中：

- `raw.jsonl`：保存参与在线认证的原始 `sensor_batch`
- `results.jsonl`：保存每个认证窗口的分数、阈值、接受/拒绝、是否打断等结果

### 5.3 预处理与训练产物

当前目录组织比较完整：

- `data_storage/processed_data/<user>/{train,val,test}.csv`
- `data_storage/processed_data/z-score/<user>/{train,val,test}.csv`
- `data_storage/processed_data/z-score/<user>/scaler.json`
- `data_storage/processed_data/window/<window_size>/<user>/{train,val,test}.csv`
- `data_storage/models/<user>/training_state.json`
- `data_storage/models/<user>/training_summary.json`
- `data_storage/models/<user>/best_lock_policy.json`
- `data_storage/models/<user>/checkpoints/*.pt`

### 5.4 合理性分析

**优点：**

- 文件系统结构直观，便于定位数据与模型；
- JSONL/CSV 对调试友好；
- 原始数据、推理数据、训练数据、模型数据分层清晰；
- 小规模原型阶段非常方便排障。

**问题：**

1. **文件存储扩展性有限。**
   - 高并发、大规模设备下，文件系统会成为瓶颈；
   - 缺少索引、事务、查询能力。

2. **原始数据选择依赖文件名或 mtime 排序，稳定性一般。**
   - 预处理优先按文件名中的时间戳排序；
   - 如果 session 文件名本身不带时间戳（当前 gRPC 场景大多是 UUID），就会退回使用文件 mtime；
   - 一旦数据被复制、恢复、迁移，mtime 可能不再代表真实采集顺序。

3. **策略文件与模型文件大量写入绝对路径，迁移性较差。**
   - 一旦部署目录变化、仓库改名、容器挂载点改变，旧策略里的 checkpoint 路径可能直接失效。

## 6. 数据预处理流程分析

核心代码：`src/processing/pipeline.py`

这是整个项目里非常关键的一段，因为它决定了训练数据如何从“原始传感器包”变成“可学习的窗口样本”。

### 6.1 预处理功能清单

预处理阶段包含以下功能：

1. 判断某用户是否达到最小训练数据量；
2. 从原始 session 文件中选择一部分 session 参与处理；
3. 从 JSON/`sensor_batch` 中抽取三类传感器记录；
4. 统一重采样到 100Hz；
5. 拼成结构化时序表；
6. 划分 train / val / test；
7. 加入 HMOG 攻击者样本到 val / test；
8. 计算 z-score 标准化参数；
9. 生成滑窗 CSV；
10. 为训练脚本准备最终窗口数据集。

### 6.2 实际处理步骤

#### 6.2.1 原始 session 选择

规则：

- 先统计某个用户所有 raw session 文件的总大小；
- 若总量 `< min_total_mb`，则不处理；
- 若达到阈值，则按文件名时间戳 / mtime 升序排序；
- 从最早的 session 开始累加，直到达到 `target_total_mb`；
- 只处理这批最早 session。

默认配置来自 `ca_config.toml`：

- `processing.min_total_mb = 100`
- `processing.target_total_mb = 100`

#### 6.2.2 传感器抽取

预处理只认三类传感器：

- 加速度计 `acc`
- 陀螺仪 `gyr`
- 磁力计 `mag`

支持从两种来源抽取：

- HTTP JSON 的 `sensor_data`
- gRPC protobuf 解出来的 `sensor_batch.samples`

并统一整理成：

- `timestamp`
- `x`
- `y`
- `z`

#### 6.2.3 重采样与对齐

系统会：

- 收集三种传感器的所有时间戳；
- 以 `sampling_rate_hz=100` 计算 10ms 基准网格；
- 对每种传感器做 `resample(...).mean().interpolate(...)`；
- 最终得到统一时轴上的 9 个轴特征：
  - `acc_x/y/z`
  - `gyr_x/y/z`
  - `mag_x/y/z`

这里还有一个值得肯定的细节：

- 代码明确处理了 pandas `resample()` 默认按 epoch 对齐的问题；
- 会把起止时间向 10ms 网格 floor/ceil，避免整个 session 重采样后全是 NaN。

#### 6.2.4 数据集切分

系统将单用户时序按长度切为：

- train: 75%
- val: 12.5%
- test: 12.5%

注意：**这是按行数/时间长度切，不是按 session 切。**

#### 6.2.5 HMOG 攻击者样本拼接

为了让 val/test 包含“非本人”样本，系统会从 HMOG 数据集中选攻击者：

- val 使用按目录排序后的第 1~10 个 HMOG 用户；
- test 使用第 11~20 个 HMOG 用户；
- 每个攻击者只保留 session 1~6；
- 读入后做列名规范化、单位转换，再拼接到 val/test。

#### 6.2.6 标准化与滑窗

系统基于 train split 计算每个特征的：

- mean
- std

写入 `scaler.json`，然后对 train/val/test 统一做 z-score。

之后再按配置生成滑窗：

- 默认窗口：`0.2s`
- overlap：`0.5`
- 采样率：`100Hz`
- 因此默认一个窗口约 20 点，步长约 10 点。

生成的窗口 CSV 仍保留：

- `subject`
- `session`
- `timestamp`
- 9 个轴
- `window_id`

### 6.3 预处理逻辑是否合理

**合理的地方：**

1. **整体流程完整。**
   - 从原始包到训练窗口的闭环是打通的。

2. **重采样思路正确。**
   - 多传感器时序确实需要统一时轴，100Hz 对手机传感器也比较常见。

3. **z-score 基于 train 计算并应用到 val/test，这一点合理。**

4. **窗口生成显式保证不跨 session。**
   - 后续训练窗口不会把不同 session 混成同一个样本。

5. **val/test 引入 HMOG 攻击者样本是合理的工程折中。**
   - 当线上当前用户只有“本人数据”时，外部公开攻击者集可以帮助构造二分类评估。

### 6.4 预处理流程存在的问题

**问题 1：切分方式会把同一 session 切进 train/val/test，存在明显数据泄漏风险。**

代码注释已经承认“可能切在 session 内部”。这在研究原型里可以接受，但在认证评估里会带来偏乐观结果：

- train 和 val/test 可能共享同一 session 的相邻时间片段；
- 这会让验证集和测试集更像“同一次录制的延续”，而不是“未见过的新会话”；
- 最终指标可能高估真实性能。

**问题 2：预处理强依赖三种传感器全部存在，缺一个就丢整段 session。**

`_resample_records()` 中，如果某个 session 缺 `acc` / `gyr` / `mag` 任意一类，直接返回 `None`。

这在模型输入固定为 9 轴时是可以理解的，但实际移动端环境中：

- 磁力计缺失、暂停、权限限制、采样异常都很常见；
- 这种严格要求会导致很多 session 无法利用；
- 数据利用率可能偏低。

**问题 3：只使用“最早满足目标大小”的 session，容易训练出陈旧模型。**

这与当前需求一致，但从持续认证的长期演化看不够理想：

- 更早的数据不一定代表当前用户行为；
- 用户行为会随时间漂移；
- 如果不使用最近数据，模型会越来越“旧”。

**问题 4：HMOG 攻击者选择策略写死。**

当前固定为：

- val：第 1~10 个 HMOG 用户
- test：第 11~20 个 HMOG 用户

这保证了可复现性，但也带来问题：

- 不够灵活；
- 对 HMOG 目录结构敏感；
- 当 HMOG 数据不完整时容易出错；
- 攻击者分布代表性有限。

**问题 5：预处理依赖本地/外挂 HMOG 目录，训练不是完全自足。**

如果 HMOG 数据不存在，尽管代码会尝试 fallback 到仓库里的 `data_storage/hmog_preprocessed` 或 `tmp_hmog`，但总体上它仍是一个外部依赖。

## 7. 模型训练流程分析

核心代码：

- 训练编排：`src/training/manager.py`
- 训练执行：`src/training/runner.py`
- 实际训练脚本：`ca_train/hmog_vqgan_experiment.py`
- 模型结构：`ca_train/vqgan.py`、`ca_train/encoder.py`、`ca_train/decoder.py`、`ca_train/codebook.py`

### 7.1 训练流程有哪些功能

1. 监控用户原始数据量是否达标；
2. 达标后自动启动预处理；
3. 用处理后的窗口数据训练 VQGAN；
4. 在多个窗口尺寸上做 sweep（虽然当前配置通常只有 0.2s）；
5. 在验证集上选出最佳窗口与阈值；
6. 保存 checkpoint、训练摘要、策略文件。

### 7.2 自动训练是怎么触发的

`TrainingManager.submit_if_ready()` 的触发条件是：

- 用户 raw data 总量达到 `min_total_mb`；
- 训练状态不是 `in_progress`；
- 当前没有同用户训练任务在跑；
- 默认情况下，如果状态已经 `completed`，不会再重复训练。

这意味着训练是：

- **按用户/设备自动触发一次的**；
- 有并发信号量限制；
- 有检查间隔，避免每个包都触发扫描。

### 7.3 模型训练阶段实际做了什么

#### 7.3.1 训练数据格式

训练脚本并不是直接读 raw_data，而是读：

```text
data_storage/processed_data/window/<window_size>/<user>/{train,val,test}.csv
```

再转成模型输入形状：

- `(N, 1, 12, T)`

其中 12 个“行”由以下特征组成：

- 9 个原始轴：`acc/gyr/mag` 的 xyz
- 3 个模长：`acc_magnitude`、`gyr_magnitude`、`mag_magnitude`

#### 7.3.2 训练目标

当前训练思路是：

- **仅用合法用户的 train split 训练 VQGAN 重建模型**；
- 模型学习“本人传感器窗口长什么样”；
- 推理时用重建误差作为异常分数。

具体分数定义：

- 先计算窗口重建误差（MSE 或 L1）；
- 再取负值 `score = -error`；
- 分数越大，越接近本人；
- 分数越小，越可能是攻击者。

#### 7.3.3 阈值选择

训练脚本会在验证集上计算：

- AUC
- FAR
- FRR
- EER
- F1
- threshold

其中 `threshold` 默认使用 **验证集的 EER 阈值**。

测试集评估时，则固定使用验证集得到的阈值。

#### 7.3.4 最佳窗口选择

如果有多个窗口大小，脚本会：

- 分别训练每个窗口；
- 用验证集 AUC 最大的那个作为最佳窗口；
- 保存到 `best_windows.json`。

#### 7.3.5 输出物

训练结束后，会产生：

- `checkpoints/vqgan_user_<user>_ws_<window>.pt`
- 同名 `.json`（VQGAN 结构配置）
- `training_summary.json`
- `training_state.json`
- `best_lock_policy.json`

### 7.4 训练流程合理性分析

**合理的地方：**

1. **用 genuine-only train，val/test 混 genuine/attacker，是合理的异常检测思路。**

2. **用重建误差作为连续认证分数，工程实现简单且可解释。**

3. **用验证集选阈值、测试集固定阈值评估，基本符合机器学习流程。**

4. **训练状态显式写盘，便于服务恢复后知道用户是否训练完成。**

5. **模型参数、阈值、窗口大小都写入策略文件，便于在线推理加载。**

### 7.5 训练流程存在的问题

**问题 1：自动训练流程没有接入离线策略搜索。**

当前自动训练链路是：

- 预处理
- 训练 VQGAN
- 直接写一个 `k_rejects` 型 `best_lock_policy.json`

但项目里其实还有一个更复杂的 `policy_search` 模块，可以做：

- `(N, M)` 投票策略搜索；
- `target_window_frr` 约束；
- Pareto 前沿筛选；
- 决策时间评估。

**问题在于：自动训练完成后并不会自动跑 `policy_search`。**

也就是说：

- 自动链路生成的是“简单 k 连拒策略”；
- 更精细的投票策略搜索是另一个离线旁路工具；
- 两者没有真正形成统一训练产线。

**问题 2：训练写出的默认策略，与在线认证默认行为不一致。**

训练产出的 `best_lock_policy.json` 默认是：

- `interrupt_rule = k`
- `k_rejects > 0`
- `vote_window_size = 0`
- `vote_min_rejects = 0`

但在线认证管理器会读取 `ca_config.toml` 的：

- `vote_window_size = 7`
- `vote_min_rejects = 6`

并优先覆盖策略文件中的投票参数。结果是：

- **训练阶段以 k 连拒为默认输出；**
- **运行阶段却很可能按 7 选 6 投票来做决策；**
- 这会导致“训练得到的策略”与“线上实际使用的策略”不是同一套。

这是本项目第二个非常重要的结构性问题。

**问题 3：训练完成后默认不增量重训。**

一旦某用户状态为 `completed`，后续再来新数据，默认不会自动重训。这样会导致：

- 模型长期陈旧；
- 无法自动适应用户行为漂移；
- “持续认证”只持续在推理侧，不持续在模型更新侧。

**问题 4：路径与环境耦合偏重。**

- HMOG 默认路径是绝对路径；
- checkpoint / policy 中常保存绝对路径；
- 训练脚本对 `ca_train` 本地目录结构有依赖。

这对单机研发没问题，但对容器迁移、异机部署、备份恢复不够友好。

## 8. 在线推理流程分析

核心代码：

- `src/authentication/manager.py`
- `src/authentication/runner.py`
- `src/authentication/vqgan_inference.py`

### 8.1 在线推理流程有哪些功能

1. 检查模型是否存在；
2. 为某个设备启动认证会话；
3. 缓存模型，避免重复加载；
4. 将实时传感器批次重采样、标准化、切窗；
5. 调用 VQGAN 计算窗口分数；
6. 依据阈值与规则生成认证结论；
7. 存储推理原始输入和结果；
8. 向 App 返回 `AuthResult`。

### 8.2 在线推理的实际过程

#### 8.2.1 启动会话

`StartAuthentication` 被调用后：

- 若模型存在，则 `AuthSessionManager.start_session()` 创建状态；
- 为该用户初始化：
  - `tail_records`（保留滑窗重叠尾部）
  - `window_index`
  - `ConsecutiveRejectTracker`
  - `VoteRejectTracker`

#### 8.2.2 每次来一个 `sensor_batch` 后如何处理

对每个成功解析的批次，在线推理层会：

1. 先将该批次写到 `inference/raw.jsonl`；
2. 从 `sensor_batch.samples` 提取三类传感器记录；
3. 与会话保留的 `tail_records` 拼接；
4. 用与训练同样的逻辑做 100Hz 重采样；
5. 读取 `scaler.json` 做 z-score；
6. 切成 `(1, 12, T)` 格式窗口；
7. 用缓存的 VQGAN 模型计算分数；
8. 根据规则决定 `accept` / `interrupt`；
9. 将结果写入 `inference/results.jsonl`；
10. 通过 gRPC 流回传 `AuthResult`。

#### 8.2.3 决策规则

在线决策支持两类：

1. **K 连续拒绝**
   - 连续 `K` 个窗口被判为拒绝，则触发打断。

2. **投票规则**
   - 最近 `N` 个窗口中，若拒绝数 `>= M`，则触发打断。

当前运行时优先走投票规则，只要：

- `vote_window_size > 0`
- `vote_min_rejects > 0`

#### 8.2.4 会话重叠处理

为了让流式批次之间能拼成连续滑窗，代码会保留上一批次尾部一段数据：

- 保留长度约为 `window_size * overlap`

这使得实时切窗不至于在批次边界处断裂。

### 8.3 在线推理流程合理性分析

**合理的地方：**

1. **在线推理尽量复用了训练预处理逻辑。**
   - 这保证训练/推理特征空间基本一致。

2. **使用模型缓存是合理的。**
   - 避免每个包重复加载模型，降低时延。

3. **会话级 tail 拼接设计是正确的。**
   - 对滑窗连续认证来说，这是必须的。

4. **推理结果单独落盘，可审计、可回放。**

### 8.4 在线推理流程存在的问题

**问题 1：会话状态是按 `user_id` 保存，而不是按 `(user_id, session_id)` 保存。**

当前 `_sessions` 的 key 是 `user_id`。这意味着：

- 同一设备如果并发开启多个认证会话；
- 后来的会话会覆盖先前的会话状态；
- 系统默认假设“一个设备同一时间只会有一个认证会话”。

这在原型阶段可以接受，但不够稳健。

**问题 2：会话可能被接受，但随后因为缺少 scaler 等依赖而无法真正产出结果。**

`has_trained_model()` 只检查：

- `best_lock_policy.json` 是否能读；
- `vqgan_checkpoint` 是否存在；
- `vqgan_config` 是否存在。

但在线推理实际还依赖：

- `processed_data/z-score/<user>/scaler.json`

若 scaler 丢失，会出现：

- `StartAuthentication` 返回 accepted；
- 但 `handle_packet()` 直接 `return None`；
- App 收不到认证结果。

**问题 3：返回给 App 的分数字段语义不完全一致。**

在线内部判定时，原始分数是 `raw_score = -reconstruction_error`。

但回传给 App 的 `AuthResult.score`：

- 在投票模式下，变成了 0~1 的“恶意比例映射值”；
- 在非投票模式下，可能又是 sigmoid 归一化值；
- 而 `threshold` 却可能仍是原始分数阈值。

这会导致：

- `score` 与 `threshold` 不一定在同一量纲；
- 客户端若直接显示或比较这两个值，会产生误解。

**问题 4：在线只支持 VQGAN 打分，而离线策略搜索支持 VQGAN+Transformer，两者能力不一致。**

这是本项目第三个非常重要的问题。

项目的 `policy_search` 支持两种打分：

- `vqgan-only`
- `vqgan+transformer`

但在线认证代码：

- `AuthRunConfig` 只加载 `vqgan_checkpoint` 和 `vqgan_config`；
- 不加载 `lm_checkpoint`；
- 不真正支持 transformer 在线打分。

这意味着：

- 如果离线用 `vqgan+transformer` 找到最优策略并写入默认策略文件；
- 在线实际仍然按 **VQGAN-only** 打分；
- 阈值、评分分布、打断策略都会错位。

这是一个会直接影响线上判决正确性的严重问题。

**问题 5：VQGAN-only 的离线最优策略默认不会被线上自动消费。**

`policy_search` 在 `vqgan-only` 模式下，默认把最佳策略写到：

```text
data_storage/models/<user>/policy_search/best_lock_policy_vqgan_only.json
```

而在线默认加载的是：

```text
data_storage/models/<user>/best_lock_policy.json
```

因此：

- 即使离线已经为 VQGAN-only 搜到了更优策略；
- 线上默认也不会自动用它；
- 除非人工再做搬运/改造。

这说明“离线最优策略搜索”与“在线默认执行策略”目前并没有真正打通。

## 9. 向 App 传输交互流程分析

这一部分重点看：服务端究竟如何和 App 协作，以及哪些交互只是“定义了但没真正用起来”。

### 9.1 App 可以从服务端得到哪些信息

#### 9.1.1 初始策略 `PolicyUpdate`

通过 `GetInitialPolicy`，服务端会返回：

- `policy_id`
- `policy_version`
- `batch_interval_ms = 1000`
- `max_payload_size_bytes`
- `upload_rate_limit = 50.0`
- `compression_algorithm = LZ4`
- `batch_size_threshold = 50`
- 三种传感器采样率都设为 100Hz

#### 9.1.2 认证会话响应 `AuthSessionResponse`

通过 `StartAuthentication`，服务端会告诉 App：

- 当前是否允许开始连续认证；
- 会话 ID；
- 使用的模型版本；
- 当前窗口大小；
- 期望决策时间上限。

#### 9.1.3 流式回包 `Ack`

App 每发一个 `DataPacket`，服务端都会返回一个 `Ack`，包含：

- `packet_id`
- 服务端接收时间
- 是否成功（但当前语义偏向“是否写盘成功”）

#### 9.1.4 实时认证结果 `AuthResult`

若当前设备会话已建立，且这一批数据足够形成窗口，则服务端会再回一个 `AuthResult`，包含：

- score / threshold
- accept / interrupt
- window_id
- model_version
- message

### 9.2 当前与 App 交互的合理性

**合理的地方：**

1. **先拉策略、再开会话、再流式上传，这个分层是对的。**

2. **`Ack` 与 `AuthResult` 分离可以降低客户端耦合。**
   - App 可以先知道“包到了”，再异步等待认证结论。

3. **心跳和聚合指标上报接口有利于移动端运维。**

### 9.3 当前与 App 交互存在的问题

**问题 1：`GetInitialPolicy` 的返回几乎是硬编码，不是强配置驱动。**

当前初始策略并没有真正根据：

- 用户状态；
- 设备能力；
- 模型版本；
- 后端负载；

做动态下发，而更像一个固定模板。

**问题 2：服务端 proto 设计的“下行控制能力”没有完全实现。**

虽然定义了：

- `KeyRotationNotice`
- `EmergencyStop`
- `AnomalyConfig`

但目前主逻辑没有真正使用这些下行能力。

**问题 3：HTTP 文档与 gRPC 实现之间存在文档漂移。**

仓库内已有文档与实际代码不完全一致，例如：

- README 中还写着 gRPC 默认端口是 `8000`，但代码默认是 `10500`；
- README 中给出的 HTTP 上传路径与实际实现不一致；
- 当前真正主链路显然是 gRPC，而部分文档仍在描述旧的 HTTP 形态。

这会直接影响 App 侧接入理解。

## 10. 离线策略搜索流程分析

核心代码：`src/policy_search/runner.py`

### 10.1 这个模块做什么

它不是在线服务直接调用的，而是一个离线分析工具，用来：

1. 基于 val/test 分数数组；
2. 穷举窗口投票参数 `(N, M)`；
3. 在不同 `target_window_frr` 约束下选择阈值；
4. 计算：
   - FAR / FRR / ERR
   - 平均首次打断时间
   - 在给定秒数内首次打断的概率；
5. 输出 grid search 结果；
6. 输出 Pareto 前沿；
7. 选择一个最优策略。

### 10.2 这个模块的价值

这是本项目里**理论上最接近“连续认证策略优化”核心逻辑**的部分。因为它不只是看单窗口分数，而是真正在优化：

- 多窗口投票参数；
- 决策时间；
- 对攻击者的打断能力；
- 对合法用户误打断的控制。

从研究逻辑上说，这一部分是合理且有价值的。

### 10.3 存在的问题

问题不在模块本身，而在于它与线上系统没有完全贯通：

1. 自动训练默认不会跑它；
2. 在线认证默认也不严格按它选出的策略执行；
3. `vqgan+transformer` 搜索结果当前无法被线上完整消费；
4. `vqgan-only` 最优策略默认也不会覆盖线上默认策略文件。

换句话说：

- **离线策略搜索做得比线上实际执行更先进；**
- **但工程整合尚未完成。**

## 11. 模型本身的结构与功能定位

核心代码：`ca_train/vqgan.py`、`ca_train/encoder.py`、`ca_train/decoder.py`、`ca_train/codebook.py`

### 11.1 模型是什么

当前线上真正使用的是一个 **VQGAN 风格的离散潜变量自编码模型**，但它的用途不是生成图像，而是重建“传感器窗口图片”。

输入窗口被组织成：

- 高度 = 12（9 个轴 + 3 个模长）
- 宽度 = 时间轴点数（默认 20 或重采样后的 target width）
- 通道 = 1

因此模型把一个传感器窗口当作一张“1×12×T 的图像”处理。

### 11.2 编码器做了什么

编码器会：

- 用卷积 + 残差块提取局部时序模式；
- 在时间轴方向逐步下采样；
- 通过 codebook 做离散量化。

这意味着它尝试把用户行为窗口压缩成较稳定的离散表示。

### 11.3 解码器做了什么

解码器根据量化表示重建原始窗口。若输入是“合法用户常见模式”，重建误差应较低；若输入偏离训练分布，重建误差会更高。

### 11.4 这一设计是否合理

**合理性：**

- 对 continuous authentication 来说，“合法用户建模 + 重建误差做异常检测”是合理路线；
- 相比监督分类，genuine-only 训练更符合真实冷启动条件；
- 12×T 的“伪图像”建模方式在工程上简单直接。

**限制：**

- 模型只利用短窗口局部模式，长期上下文能力有限；
- 依赖预处理质量较高；
- 对模态缺失、传感器漂移较敏感；
- 当前线上没有把 transformer 分支真正接进来，序列级长期信息没有被线上利用。

## 13. 当前项目的主要问题清单（按优先级排序）

### P0：必须优先解决的问题

#### P0-1 协议安全设计与实现不一致

- proto 设计的是 envelope encryption；
- 实现却是固定对称密钥 AES-GCM；
- `encrypted_dek` / `dek_key_id` / `sha256` 都没真正落地。

**影响：** 安全语义不成立，客户端与服务端对协议理解不一致。

#### P0-2 训练输出策略与线上实际决策规则不一致

- 训练默认产出 k 连拒策略；
- 线上运行时默认又被配置里的投票规则覆盖。

**影响：** 离线评估和线上行为不是同一套规则，结果不可对齐。

#### P0-3 离线 `vqgan+transformer` 策略无法被线上完整执行

- 策略搜索支持 transformer；
- 在线认证并不支持 transformer 打分。

**影响：** 一旦把 transformer 搜索结果当成线上策略，线上实际判断可能完全失真。

#### P0-4 `Ack` 成功语义错误

- 当前 `Ack.success` 更接近“写盘成功”；
- 不是“可用包成功接收”。

**影响：** App 会被误导，难以正确重传或告警。

### P1：强烈建议尽快修复的问题

#### P1-1 train/val/test 可能切到同一 session 内，存在数据泄漏

**影响：** 评估结果偏乐观，线上效果可能低于离线指标。

#### P1-2 在线会话只按用户保存，无法自然支持同用户多并发会话

**影响：** 多路认证场景下会相互覆盖。

#### P1-3 HTTP 与 gRPC 双栈协议分叉，文档和实现均有漂移

**影响：** 接入成本高，后续维护难。

#### P1-4 在线会话接受条件不够严格

- 只检查 checkpoint/config，不检查 scaler 等必要推理依赖。

**影响：** 会话可能 accepted，但永远不返回有效认证结果。

#### P1-5 用户粒度与设备粒度不统一

**影响：** 后续多设备/多用户扩展会很困难。

### P2：中期优化问题

#### P2-1 策略、checkpoint 使用绝对路径，迁移性差

#### P2-2 `GetInitialPolicy` 主要是硬编码模板，缺少动态策略下发

#### P2-3 缺少 gRPC 主链路的端到端测试

当前测试更多覆盖：

- crypto
- file storage
- preprocessing
- HTTP endpoint

但对最关键的 gRPC 在线认证主链路，缺少完整集成测试。

#### P2-4 时间戳有效区间硬编码到 2030

#### P2-5 缺少模型自动迭代更新机制

## 14. 我对本项目的总体判断

### 14.1 如果把它看成“研究原型”

这是一个**完成度较高的原型**，因为它已经具备：

- App 交互协议；
- 原始数据采集接收；
- 自动预处理；
- 自动训练；
- 在线推理；
- 打断决策；
- 日志与产物管理。

从工程完整性看，这已经不是零散脚本，而是一个有明确主线的系统。

### 14.2 如果把它看成“准生产系统”

当前还不够，主要卡在四个点：

1. **协议安全落地不完整**；
2. **训练/离线搜索/线上执行不一致**；
3. **HTTP 与 gRPC 双栈分叉**；
4. **评估与真实线上行为之间存在偏差风险**。

### 14.3 最准确的定位

我认为当前最准确的定位是：

> **一个已经打通端到端闭环、但仍处于“研究原型向工程化系统过渡阶段”的连续认证服务端。**

## 15. 建议的改进方向

### 15.1 第一阶段：先做“一致性修复”

优先建议：

1. 统一线上决策规则与训练/策略搜索输出；
2. 明确只支持 `vqgan-only` 还是真的支持 `vqgan+transformer`；
3. 修正 `Ack` 语义；
4. 让 `best_lock_policy.json` 成为唯一线上真实策略来源；
5. 避免运行时再被配置文件静默覆盖关键投票参数。

### 15.2 第二阶段：再做“协议落地”

1. 真正实现 envelope decryption；
2. 使用 `encrypted_dek` / `dek_key_id`；
3. 校验 `sha256`；
4. 加入防重放机制；
5. 打通 `KeyRotationNotice`。

### 15.3 第三阶段：再做“评估与数据治理”

1. train/val/test 按 session 切分；
2. 增量训练或周期性重训；
3. 让最近数据参与模型更新；
4. 增强对缺失传感器的容错能力；
5. 增加 gRPC 主链路集成测试。

## 17. 最终结语

这个项目最值得肯定的地方，不是单个模块写得多复杂，而是它已经搭好了**持续认证系统的完整骨架**：

- 有传输协议；
- 有模型训练；
- 有在线认证；
- 有结果回传；
- 有日志与产物。

但同样也要明确：**它现在最大的短板不是“功能不够多”，而是“各模块之间还没有完全对齐”。**

如果下一步只修一个方向，我建议优先修“**离线策略、线上执行、协议安全三者一致性**”。一旦这三件事对齐，这个项目会从“可运行原型”非常快地提升到“可信工程系统”。
