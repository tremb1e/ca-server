# 持续认证服务项目可执行整改清单

> 本清单基于 `docs/项目功能文档与流程分析.md` 中的分析结果整理，目标不是再次解释问题，而是把问题转成“可以排期、可以开发、可以验收”的整改任务。
>
> 建议把本清单作为后续版本迭代的执行底稿使用。

---

## 1. 整改目标

本项目下一阶段整改的核心目标不是继续叠加功能，而是优先解决以下三类问题：

1. **一致性问题**
   - 协议定义、服务端实现、训练输出、线上判决必须统一；
2. **可用性问题**
   - App 收到的响应语义必须清晰，线上认证必须稳定、可解释、可复现；
3. **可维护性问题**
   - HTTP/gRPC、离线/在线、训练/推理之间要减少分叉，降低后续维护成本。

---

## 2. 执行原则

### 2.1 总体原则

1. **先统一主链路，再扩展功能**
   - 优先保证 gRPC 主链路正确；
   - 不建议一边新增算法一边修协议主线。

2. **先修线上真实执行路径，再修离线辅助路径**
   - 先保证线上“实际用的策略”与“离线产出的策略”一致；
   - 再去扩展 transformer 等增强能力。

3. **先修语义错误，再修性能与扩展性**
   - 比如 `Ack.success` 含义错误、策略覆盖逻辑错误，这类问题优先级高于优化吞吐量。

4. **所有整改都要附带验收标准**
   - 没有验收标准的整改，容易变成“看起来改了，但实际上没统一”。

---

## 3. 推荐整改顺序

建议按以下 4 个阶段推进：

### 阶段 A：统一线上真实行为（最高优先级）

目标：

- 明确线上到底使用什么协议、什么策略文件、什么决策规则；
- 让 App、服务端、模型输出三者语义一致。

### 阶段 B：补齐安全协议落地

目标：

- 让 proto 里的加密/密钥字段真正发挥作用；
- 完成防重放与包完整性校验。

### 阶段 C：修正训练与评估闭环

目标：

- 让 train/val/test 切分、离线评估、线上推理更可信；
- 降低离线高分、线上失效的风险。

### 阶段 D：做工程化增强

目标：

- 清理遗留接口；
- 增加更完整测试；
- 提升迁移性、可运维性。

## 5. 详细整改任务

## A. 统一线上真实行为

### A-1 统一线上策略来源

**问题**

当前至少存在三类策略来源：

1. `src/training/runner.py` 自动生成的 `best_lock_policy.json`
2. `src/policy_search/runner.py` 生成的 `policy_search/best_lock_policy_vqgan_only.json`
3. 运行时 `ca_config.toml` 又会覆盖其中一部分参数

这会导致：

- 线上到底执行哪个策略不明确；
- 离线与线上不一致；
- 排障时难以复现。

**整改目标**

- 线上认证只读取一份策略文件；
- 配置文件只提供“默认值/兜底值”，不再静默覆盖用户级已生成策略。

**推荐方案**

建议统一为：

```text
+data_storage/models/<user>/best_lock_policy.json
```

作为线上唯一权威策略文件。

无论策略来自：

- 训练默认产出；
- 还是离线 policy_search 最优结果；

最终都写回这一个位置。

**改动点**

- `src/training/runner.py`
- `src/policy_search/runner.py`
- `src/authentication/runner.py`
- `src/authentication/manager.py`

**开发步骤**

1. 定义统一策略 schema；
2. 明确哪些字段是必须字段，哪些是可选字段；
3. policy_search 完成后默认覆盖主策略文件；
4. 在线认证仅读取主策略文件；
5. 清理 `policy_search/best_lock_policy_vqgan_only.json` 的默认线上语义，仅保留为分析副本或历史产物。

**验收标准**

- 任意用户线上启动认证时，只会读取一个确定路径；
- 日志中能明确打印“当前生效策略文件路径”；
- policy_search 生成后无需人工搬运，线上自动可用。

---

### A-2 修正运行时策略覆盖逻辑

**问题**

当前 `AuthSessionManager._policy_from_config()` 会用 `ca_config.toml` 中的投票配置覆盖策略文件中的投票配置。

这会造成：

- 训练或搜索出来的策略参数不一定真正上线；
- `best_lock_policy.json` 看起来是一套规则，线上却跑另一套。

**整改目标**

- 策略文件优先级高于全局配置；
- 全局配置只在“策略字段缺失”时生效。

**改动点**

- `src/authentication/manager.py`
- `src/ca_config.py`

**开发步骤**

1. 明确优先级：`user policy > global config > hard-coded default`；
2. 若策略文件已给出 `vote_window_size` / `vote_min_rejects` / `k_rejects`，则禁止覆盖；
3. 只在策略缺字段时，从全局配置补缺；
4. 启动会话时打印最终生效的完整决策规则。

**验收标准**

- 修改 `best_lock_policy.json` 后，线上行为随之变化；
- 修改 `ca_config.toml` 不会悄悄覆盖已有用户策略；
- 日志能准确反映最终使用的是投票还是 K 连拒。

---

### A-3 修正 `Ack.success` 语义

**问题**

当前 `Ack.success` 只表示“包写盘成功”，并不表示：

- 解密成功；
- 解压成功；
- protobuf 解析成功；
- 数据可用于推理/训练。

**整改目标**

- 明确 `Ack.success` 代表“服务端成功接收并通过主处理流程”；
- 如果只做原始留档成功但主流程失败，应有不同状态码或更细分错误码。

**推荐方案**

建议将返回语义拆成两层：

1. `success=false`：包无法进入主处理链（解密/解压/解析失败）
2. `success=true`：包已通过主处理链并被系统接受
3. 若需要保留“已落盘留档”语义，可增加扩展字段，例如 `stored_for_audit=true`

**改动点**

- `src/grpc_server.py`
- `protos/sensor_data.proto`（若增加字段）
- App 客户端对应逻辑

**开发步骤**

1. 梳理当前所有失败原因；
2. 设计统一错误码枚举：
   - `DECRYPT_FAILED`
   - `DECOMPRESS_FAILED`
   - `PARSE_FAILED`
   - `STORAGE_FAILED`
   - `MODEL_NOT_READY`
   - 等等
3. 修改 `Ack.success` 赋值逻辑；
4. 客户端按错误码决定重试或告警。

**验收标准**

- 解密失败时 `Ack.success=false`；
- 仅写盘成功但解析失败时，不再被客户端当作成功包；
- 集成测试能覆盖主要错误码分支。

---

### A-4 明确线上支持的认证方法

**问题**

离线支持：

- `vqgan-only`
- `vqgan+transformer`

线上当前只真正支持：

- `vqgan-only`

**整改目标**

必须二选一，先做清晰：

1. **短期推荐方案**：明确线上只支持 `vqgan-only`，并禁止 transformer 策略进入线上默认策略；
2. **长期增强方案**：补齐 transformer 在线推理链路。

**建议路线**

推荐先走短期方案：

- 当前版本把线上能力显式收敛到 `vqgan-only`；
- 避免离线搜出 transformer 最优策略却被线上错误执行。

**改动点**

- `src/policy_search/runner.py`
- `src/authentication/runner.py`
- `src/authentication/manager.py`
- 文档与日志

**开发步骤（短期推荐）**

1. 将 `best_lock_policy.json` 线上默认写入策略限制为 `vqgan-only`；
2. 若选择 transformer 模式，只输出分析文件，不覆盖线上主策略；
3. `StartAuthentication` 时打印当前 `auth_method`；
4. 对不支持的线上策略类型直接拒绝启动会话并返回明确错误。

**验收标准**

- 线上不会加载自己不支持的策略类型；
- 任何上线策略都能被 `AuthSessionManager` 完整执行；
- App 不会收到与实际模型不匹配的阈值和分数。

---

### A-5 统一身份粒度（device vs user）

**问题**

当前代码的真实主索引基本是 `device_id_hash`，但协议又定义了 `user_id_hash`。

**整改目标**

明确当前版本到底是：

- 按设备建模；还是
- 按用户建模。

**推荐方案**

若当前 App 侧本来就是“每台设备只服务一个用户”，建议短期明确为：

- **V1 版本按 `device_id_hash` 建模和认证**；
- `user_id_hash` 暂不作为主索引，只保留扩展字段。

**改动点**

- `protos/sensor_data.proto` 注释
- `src/grpc_server.py`
- `src/processing/pipeline.py`
- 文档

**验收标准**

- 文档、日志、目录结构、模型索引全部使用同一粒度；
- 不再出现“协议看起来按 user，代码实际按 device”的混淆。

---

## B. 补齐安全协议落地

### B-1 实现真实 envelope 解密

**问题**

proto 声明了 `encrypted_dek`、`dek_key_id`，但服务端实际没有使用，仍然是固定字符串派生 AES key。

**整改目标**

- 服务端真正支持每包 DEK；
- `dek_key_id` 能定位服务端密钥版本；
- `encrypted_sensor_payload` 使用解出来的 DEK 解密。

**改动点**

- `src/crypto/decryption.py`
- `src/crypto/payload_codec.py`
- `src/grpc_server.py`
- 密钥管理模块（需要新增）
- 部署配置

**开发步骤**

1. 新增密钥解析与选择模块；
2. 用服务端私钥解密 `encrypted_dek`；
3. 使用解密出的 DEK 解 `encrypted_sensor_payload`；
4. 移除固定 `settings.encryption_key` 的主链路依赖，保留仅用于调试模式；
5. 区分生产模式与开发模式。

**验收标准**

- `encrypted_dek` 与 `dek_key_id` 在主链路被真正使用；
- 错误 key_id 会返回明确错误；
- 开发环境下仍可通过 mock key 路径调试。

---

### B-2 校验 payload 完整性

**问题**

`sha256` 字段当前只被保存，不被校验。

**整改目标**

- 收到包后对 payload 做哈希校验；
- 哈希不匹配时拒绝进入主流程。

**改动点**

- `src/grpc_server.py`
- 可新增 `src/validators` 下的完整性校验模块

**开发步骤**

1. 明确 `sha256` 校验对象是密文还是明文；
2. 服务端在落盘前完成校验；
3. 哈希不一致时返回明确错误码；
4. 记录安全告警日志。

**验收标准**

- 篡改 payload 后服务端可识别；
- 哈希失败包不会进入训练/推理链；
- 有单元测试与集成测试覆盖。

---

### B-3 增加防重放与乱序控制

**问题**

当前虽有 `packet_seq_no`、`seq_no` 字段，但未真正利用。

**整改目标**

- 检测重复包、严重乱序包、重放包；
- 保护训练数据与在线推理状态不被污染。

**改动点**

- `src/grpc_server.py`
- 可能新增轻量状态存储（内存 + 可选持久化）

**开发步骤**

1. 以 `(device_id_hash, session_id)` 为键，维护最近接收序号；
2. 若发现已处理过的 `packet_seq_no`，直接拒绝；
3. 若允许小范围乱序，定义窗口缓冲策略；
4. 定义重放错误码与日志。

**验收标准**

- 重复发送同一包时不会重复进入训练/推理；
- 在线状态不会被重复包污染；
- 日志能明确看到拒绝原因。

---

### B-4 接入密钥轮换能力

**问题**

proto 定义了 `KeyRotationNotice`，但当前未实际使用。

**整改目标**

- 让客户端可接收新公钥版本；
- 让服务端能同时支持旧新 key 的过渡期。

**改动点**

- `src/grpc_server.py`
- 密钥管理模块
- App 客户端配套逻辑

**验收标准**

- 服务端可以识别多个 key version；
- 新旧 key 过渡期可同时解包；
- 下行可通知客户端切换新 key。

---

## C. 修正训练与评估闭环

### C-1 按 session 切分 train/val/test

**问题**

当前 `_split_by_ratio()` 按时间长度直接切，可能把同一 session 切进多个 split。

**整改目标**

- train/val/test 按 session 切分；
- 评估更接近真实“未见过的新会话”。

**改动点**

- `src/processing/pipeline.py`
- 对应测试

**开发步骤**

1. 先按 session 聚合；
2. 再以 session 为单位做 75/12.5/12.5 切分；
3. 保证同一 session 只在一个 split 中出现；
4. 若 session 数量太少，定义降级策略。

**验收标准**

- 任意 session 不会同时出现在 train 和 val/test；
- 预处理测试新增“同 session 不跨 split”断言；
- 离线指标仍可正常生成。

---

### C-2 自动训练后接入策略搜索

**问题**

当前自动训练只训练 VQGAN 并输出 K 连拒策略，不会自动跑 policy_search。

**整改目标**

- 自动训练完成后，自动生成线上真正使用的策略；
- 避免“训练结束了，但线上策略还是旧的/粗糙的”。

**改动点**

- `src/training/manager.py`
- `src/training/runner.py`
- `src/policy_search/runner.py`

**开发步骤**

1. 训练完成后自动调用 policy_search（至少 `vqgan-only` 路径）；
2. policy_search 成功后覆盖主策略文件；
3. 若 policy_search 失败，则回退到训练默认策略并记录告警；
4. 将训练状态细分为：
   - `training_completed`
   - `policy_ready`

**验收标准**

- 用户达到训练阈值后，最终会自动得到一份线上可直接用的策略；
- 无需手工再跑搜索脚本；
- 训练状态日志能区分“模型有了”和“策略也准备好了”。

---

### C-3 增量/周期性重训机制

**问题**

训练一旦 `completed`，新数据默认不再触发重训。

**整改目标**

- 当新数据积累到一定程度时，自动重训或周期性重训；
- 让模型能跟随用户行为变化。

**推荐方案**

增加如下阈值之一：

1. 相比上次训练新增数据量超过 X MB；
2. 距上次训练时间超过 Y 天；
3. 在线误拒绝率/告警率异常升高时触发重训。

**改动点**

- `src/training/manager.py`
- `training_state.json` 结构
- 配置文件

**验收标准**

- 已完成训练的用户，在新增数据足够后可再次进入训练；
- 重训不会并发堆积；
- 旧模型与新模型切换过程可观测。

---

### C-4 增加缺失传感器容错

**问题**

当前缺少任一传感器即整段 session 被丢弃。

**整改目标**

- 提高线上数据可利用率；
- 减少因磁力计异常导致整个 session 失效。

**可选方案**

1. 短期：允许某些传感器缺失时用 NaN/0 或插值回填；
2. 中期：训练多模态子模型；
3. 长期：支持动态传感器掩码输入。

**验收标准**

- 单一传感器缺失不再导致整段 session 被直接放弃；
- 训练/推理能明确知道当前使用了哪些模态。

---

## D. 工程化增强

### D-1 明确 HTTP 接口地位

**问题**

HTTP 与 gRPC 协议已明显分叉，HTTP 更像遗留入口。

**整改目标**

必须明确二选一：

1. 保留 HTTP，仅作为调试/兼容接口；
2. 或正式弃用 HTTP 上传主链。

**推荐方案**

建议：

- 正式声明 gRPC 为主链路；
- HTTP 保留为调试/回放入口；
- 在文档中清晰标注“不参与在线认证主闭环”。

**改动点**

- `README.md`
- `server.md`
- `src/api/sensor_data.py`
- 测试说明

**验收标准**

- 新接入方看文档不会混淆主入口；
- 不再把 HTTP 当成线上主要上传协议。

---

### D-2 增加 gRPC 端到端测试

**问题**

当前最核心的 gRPC 主链路缺少完整集成测试。

**整改目标**

至少覆盖：

1. `StartAuthentication` 正常接受/拒绝；
2. `StreamSensorData` 的 Ack 与 AuthResult；
3. 解密失败、解析失败、模型未就绪；
4. 在线推理结果回传；
5. 训练触发检查。

**改动点**

- `tests/integration/`

**验收标准**

- 关键主链路具备自动回归能力；
- 后续修协议/修策略时可以快速发现破坏性改动。

---

### D-3 去绝对路径化

**问题**

策略、checkpoint、部分默认路径存在绝对路径依赖。

**整改目标**

- 策略文件内尽量保存相对路径或可重定位路径；
- 容器/宿主迁移后不需要手工改 JSON。

**改动点**

- `src/training/runner.py`
- `src/policy_search/runner.py`
- `src/authentication/runner.py`

**验收标准**

- 改变项目根目录后，模型策略仍能被成功解析；
- 只需修改少量配置，不需批量重写历史策略文件。

---

### D-4 动态初始策略下发

**问题**

`GetInitialPolicy` 当前基本是硬编码模板。

**整改目标**

让它至少能根据：

- 服务器配置；
- 用户状态；
- 当前版本；
- 流控需求；

动态下发策略。

**改动点**

- `src/grpc_server.py`
- 配置文件

**验收标准**

- 修改配置后，下发策略同步变化；
- 不同用户/版本可以返回不同策略；
- 不需要改代码才能调整采样率与上传节奏。

## 7. 我最推荐先做的 6 个任务

如果当前资源有限，我建议先按下面顺序做：

1. **A-1 统一线上策略来源**
2. **A-2 修正运行时策略覆盖逻辑**
3. **A-3 修正 `Ack.success` 语义**
4. **A-4 明确线上仅支持 `vqgan-only`**
5. **C-1 按 session 切分 train/val/test**
6. **D-2 增加 gRPC 端到端测试**

原因是：

- 这 6 项改动虽然不一定最“炫”，但它们最直接影响系统真假一致性；
- 做完后，线上行为会立刻变得更可信；
- 后续再做安全协议和能力扩展，风险会小很多。

## 9. 最终建议

本项目当前最需要的不是“继续加新模型”，而是：

- **先让主链路说同一种语言**；
- **先让线上真正执行离线选出来的策略**；
- **先让协议定义和实现匹配**。

只有这三件事完成后，后续无论是继续上 transformer、继续做更复杂策略搜索、还是继续做多设备多用户支持，才会建立在稳固的基础之上。
